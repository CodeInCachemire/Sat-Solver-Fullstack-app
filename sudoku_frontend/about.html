Behind The Solver

A production-grade architecture combining boolean satisfiability algorithms with real-time constraint validation and async job queue processing.

Project Overview

Sudoku SAT Solver: Full-Stack Architecture

This project demonstrates a modern full-stack implementation of a Sudoku solver using SAT (Boolean Satisfiability) algorithms. Rather than using traditional constraint propagation, we encode Sudoku puzzles as boolean formulas and leverage industrial-strength SAT solving techniques to determine solvability and compute solutions.

Why SAT? SAT solving is NP-complete, making Sudoku solving a legitimate application of formal methods. This approach demonstrates understanding of computational complexity, algorithm design, and production system architecture.

Core Algorithm: DPLL

Davis-Putnam-Logemann-Loveland Algorithm

The solver implements the DPLL algorithm, a foundational SAT solving technique that combines:

  • Unit Propagation: Eliminate variables forced to a single truth value
  • Pure Literal Elimination: Remove variables appearing with only one polarity
  • Backtracking Search: Recursively try assignments, backtrack on conflicts
  • Conflict Detection: Prune impossible branches early
Performance Characteristic: While SAT is NP-complete in general, well-engineered DPLL solvers with heuristics achieve ~2-5 seconds per puzzle even for complex instances. Modern heuristics like Minimum Remaining Values (MRV) and Conflict-Driven Clause Learning (CDCL) further optimize search.

Boolean Formula Encoding: Tseitin Transform

Sudoku constraints are not naturally expressed as SAT problems. We use Tseitin encoding to convert higher-order logical constraints into Conjunctive Normal Form (CNF), the canonical input format for SAT solvers.

Encoding Process

  1. Variable Creation: Create 729 boolean variables (9×9 grid × 9 possible values)
  2. Constraint Generation: Encode Sudoku rules as CNF clauses:
    - Row uniqueness: ∀ row, at most one value per cell
    - Column uniqueness: ∀ column, at most one value per cell
    - Box uniqueness: ∀ 3×3 box, at most one value per cell
  3. Pre-filled Cells: Add unit clauses for given clues
  4. SAT Formula: Conjunction of all clauses (AND of all constraints)

Result: ~350-400 clauses depending on puzzle, solved via DPLL without backtracking for valid puzzles.

Backend Architecture: Distributed Job Queue

Tech Stack

API Layer

FastAPI (Python) with CORS middleware and structured error handling

Compute Engine

C backend with optimized DPLL, Tseitin encoding, CNF parsing

Job Queue

Redis with sync workers and job status tracking

Database

PostgreSQL for job metadata, results, and deduplication

Async Pipeline Design

User Request
↓
FastAPI Endpoint (validation, deduplication check)
↓
PostgreSQL (check if formula solved before)
↓
[Cache Hit] → Return cached result instantly
[Cache Miss] → Queue job in Redis
↓
Return job_id to client (<100ms)
↓ (async)
Background Worker pulls from queue
↓
C SAT Solver executes (2-5 seconds)
↓
PostgreSQL stores result + variable assignments
↓
Client polls /jobs/{id} for result

Key Design Features

  • Deduplication: Same formula submitted twice retrieves cached result
  • Job Tracking: Full lifecycle visibility (queued → processing → completed)
  • Non-blocking API: API returns immediately; processing happens async
  • Horizontal Scaling: Multiple workers can process queue independently
  • Error Resilience: Failed jobs logged with detailed diagnostics
  • Production Ready: CORS headers, input validation, rate-limiting compatible

Frontend: Real-Time Constraint Validation

The web interface provides immediate feedback on user input using client-side constraint validation, eliminating wasted API calls and improving UX.

Real-Time Validation Engine

  • Conflict Detection: As user enters values, solver highlights duplicates in row/column/box
  • Live Feedback: Invalid cells marked instantly (no network latency)
  • Smooth UX: Users know immediately if their move violates rules
  • Image Upload: Parse Sudoku images using OCR-like analysis

Technical Implementation

Built with vanilla JavaScript (no frameworks) for minimal dependencies. Grid representation stored as 2D array with efficient O(1) conflict checking via Set-based lookups.

Performance & Optimization

C Backend Optimizations

  • Memory Efficiency: Stack-based recursion for DPLL with bounds checking
  • Clause Indexing: HashMap of variable occurrences for fast propagation
  • Early Termination: Detect unsatisfiability without exploring full tree
  • Variable Ordering Heuristics: MRV (Minimum Remaining Values) for smart branching

API & Queue Optimization

  • Caching Strategy: PostgreSQL UNIQUE constraint on normalized formulas
  • Job Pooling: Reuse worker processes; avoid spawn overhead
  • Result Memoization: Lookup solved puzzles in ~5-10ms
  • Async I/O: Non-blocking database and Redis calls (v2 roadmap)
Measured Performance: API response time <100ms for cache hits, ~3 seconds end-to-end for new puzzles (2-5 seconds solver + overhead).
